# Lessons

- Auto demos must be idempotent across reconnects. If a demo depends on a known board state, reset the board explicitly before the first scripted move instead of assuming a fresh session.
- LLM tool handlers must validate missing or malformed arguments defensively. Even when a schema marks fields as required, models can still emit partial tool calls during real sessions.
- Never share a Pipecat `LocalSmartTurnAnalyzerV3` (or any turn analyzer) instance across concurrent voice sessions to save model-load cost. It holds per-connection mutable state (`_audio_buffer`, `_speech_triggered`, `_silence_ms`); sharing it mixes audio buffers between unrelated callers. The ONNX model load cost is inherent per-connection with this Pipecat version — there's no clean seam to share just the inference session without forking the library.
- `pipecat.processors.transcript_processor.TranscriptProcessor` is marked deprecated (in favor of `LLMUserAggregator`/`LLMAssistantAggregator` events) but is still the simplest correct way to get real user+assistant transcript text with roles/timestamps out of a running pipeline. Used via `transcript.user()`/`transcript.assistant()` inserted around the STT/TTS stages, with `on_transcript_update` bridging into app-level conversation state. Revisit if a future Pipecat major version removes it.
- Any tool registered on a Pipecat LLM (`llm.register_function`) must never let an exception escape without calling `params.result_callback(...)`. An uncalled callback hangs the assistant's turn silently — there's no timeout that recovers it. Wrap every tool body so `BoardCommandError` (or any other failure) turns into a compact `{"error": {...}}` payload the LLM can react to instead.
- When a Playwright WebSocket-dependent e2e test hangs at "connecting" in a sandboxed local dev container (plain `fetch()` works, `new WebSocket()` never opens or errors, stays in readyState 0), suspect the Chromium sandbox itself before suspecting app code — verify by connecting with a raw `websockets` Python client directly against the same backend port. In one observed case here it resolved itself after one warm run in the same session; if it doesn't, retry once with `launchOptions: { args: ["--no-sandbox"] }` for local diagnosis only, never commit that flag (CI's `ubuntu-latest` runners don't need it).
